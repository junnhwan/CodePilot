# CodePilot 剩余工作清单

## 1. 文档目的

这份文档用于回答一个简单问题：

在 `P13` 已完成之后，CodePilot 还剩哪些工作值得继续做，哪些应该后置，哪些当前明确不做。

本文档基于以下仓库真相整理：

- 当前分支：`phase/P13`
- 当前阶段状态：`P13 DONE`
- 已完成范围：`P1 ~ P13`
- 当前时间锚点：`2026-04-24`

---

## 2. 当前结论

当前主链最小闭环已经完成：

- `Diff接收 -> Planning -> Context编译 -> Multi-Agent Review -> Merge -> Report -> Memory注入`
- `Gateway / CLI / MCP Server / Eval Center` 均已有最小可运行闭环
- `P13` 已补齐最小 baseline 对比、稳定 scorecard / report 输出、可执行评测入口

但距离最初设计文档里的“完整版 CodePilot”，仍有几块高价值工作没有完成。

---

## 3. 必须继续补齐

下面这些能力仍属于主线收尾，建议继续完成。

### 3.1 Dream 记忆沉淀最小闭环

- [ ] 设计并实现 `ReviewResult -> MemoryPlan -> ProjectMemory 更新` 的最小 Dream 主链
- [ ] 补齐 Dream Phase 1：从本次 `ReviewResult` 中识别可沉淀的 `ReviewPattern` / `TeamConvention`
- [ ] 补齐 Dream Phase 2：把增量结果写回 `ProjectMemoryRepository`
- [ ] 明确 `REVIEW_COMPLETED != MEMORY_UPDATED` 的状态边界和失败处理
- [ ] 为 Dream 增加最小可验证测试，覆盖成功沉淀、空沉淀、异常失败和幂等行为

完成标准：

- review 完成后能独立触发 Dream
- Dream 失败不会伪装成 review 成功后“已更新记忆”
- 下一次 review 能观察到被沉淀后的记忆效果

### 3.2 Project Memory 检索能力增强

- [ ] 在现有 P9 最小 recall 基础上，补齐更稳定的检索排序与召回策略
- [ ] 评估并接入 `PgVector` 路径，把当前词法召回升级为“结构化过滤 + 向量检索”的可选实现
- [ ] 保持 `ProjectMemory` 概念不变，不额外发明平行概念
- [ ] 增加检索命中率、召回稳定性和错误定位的测试

完成标准：

- 检索能力仍保持 `core` 内聚，不为数据库多造一层空心抽象
- 没有 `PgVector` 时仍可降级运行
- 有 `PgVector` 时能证明 recall 质量或排序质量确实提升

### 3.3 Global Knowledge 最小可用能力

- [ ] 明确 `Global Knowledge` 与 `ProjectMemory` 的边界
- [ ] 先收敛一版只读通用知识源，不混入 Dream 编辑能力
- [ ] 支持在 `ContextPack` 中按任务类型注入有限的通用安全 / 性能知识
- [ ] 补齐“项目记忆优先，通用知识兜底”的选择策略

完成标准：

- reviewer 能区分“项目内已知模式”和“通用安全 / 性能知识”
- 不会因为引入通用知识把 prompt 膨胀到失控

### 3.4 评测场景从最小 pack 扩到阶段目标规模

- [ ] 把当前 `minimal-scenario-pack` 扩成更接近 `04-评测与交付计划.md` 的场景集
- [ ] 逐步补齐安全、性能、规范、可维护性、无问题变更、跨文件影响、大 PR 等类别
- [ ] 为每个场景补充稳定 ground truth，避免“跑通了但无法解释”
- [ ] 保持现有 `EvalScenario -> EvalRunner -> Scorecard` 主链，不重写第二套评测框架

完成标准：

- 场景覆盖面明显扩大
- 每个场景都能解释“为什么 pass / fail / error”
- 统计口径在多次运行中保持稳定

### 3.5 Baseline 体系继续补齐

- [ ] 在当前 `CodePilot / Direct LLM / Full Context LLM` 基础上，评估是否追加 `Lint Only`
- [ ] 如果补 `Lint Only`，优先用最小可运行方案，不引入大规模平台化静态分析编排
- [ ] 为 baseline 对比补齐更可读的结论摘要
- [ ] 明确哪些 baseline 只是演示性对比，哪些 baseline 作为长期回归基线

完成标准：

- baseline 数量增加后，报告仍然可读
- 每路 baseline 的输入边界和统计口径都清楚可解释

### 3.6 Session 恢复链路增强

- [ ] 在 P11 最小 checkpoint + replay 基础上，补强更多中断场景
- [ ] 覆盖 `PLANNING / REVIEWING / MERGING / REPORTING` 不同中断点的恢复测试
- [ ] 明确超时、崩溃、部分 task 已完成时的恢复行为
- [ ] 检查 `SessionEvent` 载荷是否足够支持后续 Dream / Report 继续推进

完成标准：

- 恢复后不会重复执行已完成 task
- 恢复后 merge / report 的输入结果完整
- 错误信息能直接定位到 `sessionId / taskId / agentName`

## 6. 建议执行顺序

如果按价值和依赖关系排序，建议顺序如下：

1. `Dream` 最小闭环
2. `ProjectMemory + Global Knowledge` 检索增强
3. 扩完整 eval scenario pack
4. 补 `Lint Only` 或其他低成本 baseline
5. 增强 Session 恢复链路
6. README / 简历 / 面试 / 博客材料

---

## 7. 一句话总结

当前 CodePilot 已经完成了主链最小闭环，后续最值得继续做的不是再铺新战线，而是把 `Memory 沉淀`、`检索增强`、`评测扩容` 和 `恢复增强` 这四块补实。
