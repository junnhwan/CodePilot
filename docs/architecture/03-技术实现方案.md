# CodePilot — 技术实现方案

---

## 1. 技术栈

| 层面 | 技术选型 | 理由 |
|------|---------|------|
| **语言** | Java 21 | 虚拟线程（Loom）天然适合 Agent 并发调度 |
| **框架** | Spring Boot 3.5+ | Web 层 + 依赖注入，不用于 Agent 内核 |
| **构建** | Maven 多模块 | 模块边界清晰 |
| **AST 解析** | JavaParser 3.26+ | 成熟的 Java AST 解析库，支持 Java 21 |
| **Git 操作** | JGit 7.x | 纯 Java Git 实现，clone/diff/blame/log |
| **LLM 调用** | 自封装 + OpenAI 兼容协议 | 不依赖 Spring AI / LangChain4j 的高层封装 |
| **向量存储** | PgVector (PostgreSQL 扩展) | Memory 检索，一个 PG 搞定关系+向量 |
| **关系存储** | MySQL 8.x | Session、Finding、Report 等结构化数据 |
| **缓存** | Redis | 工具调用结果缓存、会话锁、限流 |
| **消息队列** | Redis Stream（轻量） | Webhook 事件缓冲，解耦 Gateway 和 Core |
| **前端** | 后期考虑，MVP 用 CLI + PR Comment | 不作为首期重点 |
| **容器化** | Docker + docker-compose | 一键启动，面试演示用 |

### 1.1 为什么不用 Spring AI / LangChain4j 的高层封装

简历项目的核心价值在于**展示工程思考**，而不是"会调框架 API"。具体来说：

- **Agent Loop** 自己写 → 可以聊 ReAct 循环、决策解析、Loop Detection、Context Governance
- **Tool 调度** 自己写 → 可以聊并发分区、结果缓存、超时控制
- **Memory** 自己写 → 可以聊分层记忆、Dream 沉淀、向量检索 + 规则匹配混合策略
- **Context 编译** 自己写 → 可以聊 AST 感知的上下文选择、token 预算分配

如果用框架封装，面试官问"你的 Agent Loop 怎么实现的"，回答"调了 LangChain4j 的 AiServices"——这就没有区分度了。

底层 LLM 调用可以用 Spring AI 的 `ChatModel` 做薄封装（只做 HTTP 调用 + 流式解析），但 Agent 编排层全部自己实现。

## 2. 关键组件详细设计

### 2.1 LLM 调用层

```java
// 统一的 LLM 调用接口
public interface LlmClient {
    LlmResponse chat(LlmRequest request);
    Flux<LlmChunk> stream(LlmRequest request);
}

record LlmRequest(
    String model,
    List<Message> messages,
    List<ToolDefinition> tools,     // 可选：function calling
    Map<String, Object> params      // temperature, max_tokens 等
) {}

record LlmResponse(
    String content,
    List<ToolCallInResponse> toolCalls,  // 模型返回的工具调用
    Usage usage,
    String finishReason
) {}
```

**双模式工具调用**：
- **Function Calling 模式**：模型原生支持（GPT-4、Qwen-Max、DeepSeek-V3）→ 直接用
- **Prompt 驱动模式**：模型不支持（部分开源模型）→ 在 system prompt 中注入工具描述，解析模型输出中的结构化标记（借鉴 ThoughtCoding）

```java
class ToolCallParser {
    // Function Calling 模式：直接从 response.toolCalls 解析
    // Prompt 驱动模式：从 response.content 中解析 ```tool_call json 块
    List<ParsedToolCall> parse(LlmResponse response, ToolCallMode mode);
}
```

**重试策略**（借鉴 nanobot）：
- 429 分类：retryable（速率限制）vs non-retryable（配额耗尽）
- Retry-After 解析：从 header 和 body 两个来源
- 指数退避 + 抖动
- 图像剥离降级（多模态请求 token 超限时去除图像重试）

### 2.2 Agent Loop 实现

```java
public class ReviewEngine {

    private final LlmClient llm;
    private final ToolExecutor toolExecutor;
    private final ContextGovernor contextGovernor;
    private final LoopDetector loopDetector;
    private final int maxIterations;
    private final int tokenBudget;

    public ReviewTaskResult execute(ReviewTask task, ContextPack context) {
        // 1. 构建初始 prompt
        var messages = new ArrayList<Message>();
        messages.add(systemPrompt(task, context));
        messages.add(userPrompt(task, context));

        var findings = new ArrayList<Finding>();
        var iterations = 0;

        while (iterations < maxIterations) {
            iterations++;

            // 2. 调用 LLM
            var response = llm.chat(buildRequest(messages, task.availableTools()));
            var decision = parseDecision(response);

            switch (decision.type()) {
                case TOOL_CALL -> {
                    // 3. 执行工具
                    var results = toolExecutor.executeAll(decision.toolCalls());
                    // 4. 工具结果追加到上下文
                    messages.add(assistantMessage(response));
                    messages.addAll(results.toMessages());

                    // 从响应中提取 partial findings
                    findings.addAll(extractPartialFindings(response));
                }
                case NEED_CONTEXT -> {
                    // 5. 按需编译更多上下文
                    var extra = contextCompiler.compileMore(decision.contextRequest());
                    messages.add(assistantMessage(response));
                    messages.add(toolResultMessage("context_expansion", extra.summary()));
                }
                case DELIVER -> {
                    // 6. Agent 认为审查完成
                    findings.addAll(extractFindings(response));
                    return ReviewTaskResult.success(task, findings);
                }
            }

            // 7. Loop Detection
            if (loopDetector.isStuck(messages)) {
                return ReviewTaskResult.degraded(task, findings,
                    "Agent loop detected, returning partial findings");
            }

            // 8. Token 预算检查 + Context 治理
            var tokenCount = tokenCounter.count(messages);
            if (tokenCount > tokenBudget) {
                messages = contextGovernor.compact(messages, tokenBudget);
            }
        }

        return ReviewTaskResult.timeout(task, findings);
    }
}
```

### 2.3 Context Governance 实现

```java
public class ContextGovernor {

    private final TokenCounter tokenCounter;
    private final int reserveBudget;  // 为新内容预留的 token 数

    public List<Message> compact(List<Message> messages, int budget) {
        var currentTokens = tokenCounter.count(messages);
        if (currentTokens <= budget) return messages;

        var target = budget - reserveBudget;

        // 策略 1: Microcompact — 旧的工具调用结果摘要化
        messages = microcompact(messages);
        if (tokenCounter.count(messages) <= target) return messages;

        // 策略 2: History Snipping — 裁剪最老的交互轮次
        // 保证在合法消息边界上裁剪（不破坏 tool_call + tool_result 配对）
        messages = historySnip(messages, target);
        if (tokenCounter.count(messages) <= target) return messages;

        // 策略 3: Orphan Cleanup — 清理失去配对的孤立消息
        messages = orphanCleanup(messages);

        return messages;
    }

    private List<Message> microcompact(List<Message> messages) {
        // 遍历消息，将超过 N 轮的 tool_result 内容替换为摘要
        // 摘要用小模型生成（或规则提取：取前 200 字 + 结论）
        return messages.stream().map(msg -> {
            if (msg.role() == TOOL && msg.age() > OLD_THRESHOLD) {
                return msg.withContent(summarize(msg.content()));
            }
            return msg;
        }).toList();
    }

    private List<Message> historySnip(List<Message> messages, int target) {
        // 从最老的消息开始删除，直到 token 数 <= target
        // 关键：不破坏 tool_call + tool_result 配对
        // 实现：按"轮次"（一轮 = assistant + tool_calls + tool_results）删除
        var rounds = groupIntoRounds(messages);
        while (tokenCounter.count(messages) > target && rounds.size() > 1) {
            rounds.remove(0);  // 移除最老的一轮
            messages = flattenRounds(rounds);
        }
        return messages;
    }
}
```

### 2.4 Loop Detection 实现

```java
public class LoopDetector {

    private final LlmClient smallModel;  // 用小模型做判断
    private final int patternThreshold = 3;

    public boolean isStuck(List<Message> messages) {
        // 模式 1: 重复工具调用检测
        if (detectRepeatedToolCalls(messages)) return true;

        // 模式 2: 重复内容检测
        if (detectRepeatedContent(messages)) return true;

        // 模式 3: LLM 判断（仅在模式 1/2 触发后执行，节省 token）
        if (detectViaLlm(messages)) return true;

        return false;
    }

    private boolean detectRepeatedToolCalls(List<Message> messages) {
        // 取最近 5 轮，检查是否连续 3 次调用相同工具 + 相同参数
        var recentCalls = extractRecentToolCalls(messages, 5);
        return recentCalls.stream()
            .collect(Collectors.groupingBy(ToolCall::signature, Collectors.counting()))
            .values().stream()
            .anyMatch(count -> count >= patternThreshold);
    }

    private boolean detectViaLlm(List<Message> messages) {
        // 构造判断 prompt：最近几轮对话，判断 Agent 是否陷入循环
        var prompt = """
            以下是一个 AI Code Review Agent 的最近几轮对话。
            请判断 Agent 是否陷入了无效循环（重复相同的操作而没有进展）。
            只回答 YES 或 NO。
            """;
        var response = smallModel.chat(prompt, recentMessagesSummary(messages));
        return response.trim().equalsIgnoreCase("YES");
    }
}
```

### 2.5 AST 工具实现

```java
public class AstParseTool implements Tool {

    private final JavaParserCache parserCache;  // 解析结果缓存

    @Override
    public ToolResult execute(ToolCall call, ToolContext ctx) {
        var filePath = (String) call.arguments().get("file_path");
        var parseMode = ParseMode.valueOf((String) call.arguments().getOrDefault("mode", "FULL"));

        var result = parserCache.parse(filePath);

        return switch (parseMode) {
            case FULL -> ToolResult.success(toFullAst(result));
            case SYMBOLS -> ToolResult.success(toSymbolTable(result));
            case METHOD_SIG -> ToolResult.success(toMethodSignatures(result));
            case CALL_GRAPH -> ToolResult.success(toCallGraph(result));
        };
    }

    record AstResult(
        String filePath,
        List<ClassInfo> classes,      // 类名、父类、接口
        List<MethodInfo> methods,     // 方法签名、返回类型、参数
        List<FieldInfo> fields,       // 字段名、类型
        List<CallEdge> callEdges,     // 调用关系
        List<Reference> references    // 被引用关系
    ) {}
}
```

## 3. 接口设计

### 3.1 Webhook 接收

```
POST /api/v1/webhook/github
  Header: X-Hub-Signature-256 (HMAC 验签)
  Body: GitHub Webhook Payload

处理流程：
1. 验签
2. 解析 PR 事件（opened / synchronize / reopened）
3. 去重（同一 PR + commit 只处理一次）
4. 推入 Redis Stream
5. 返回 200
```

### 3.2 SSE 进度推送

```
GET /api/v1/reviews/{sessionId}/stream
  Accept: text/event-stream

事件类型：
  session_created   → {"sessionId": "...", "prUrl": "..."}
  plan_ready        → {"plan": {...}, "taskCount": 5}
  task_started      → {"taskId": "...", "type": "SECURITY", "files": [...]}
  finding_found     → {"finding": {...}, "severity": "HIGH"}
  task_completed    → {"taskId": "...", "findingCount": 3}
  review_completed  → {"result": {...}, "reportUrl": "..."}
  review_failed     → {"error": "...", "partialFindings": [...]}
```

### 3.3 查询接口

```
GET  /api/v1/reviews/{sessionId}           → ReviewSession 详情
GET  /api/v1/reviews/{sessionId}/report    → Markdown 报告
GET  /api/v1/projects/{projectId}/memory   → 项目记忆列表
POST /api/v1/projects/{projectId}/conventions  → 手动添加团队规范
GET  /api/v1/stats                          → 全局统计（review 数量、平均耗时等）
```

## 5. Prompt Engineering

### 5.1 System Prompt 模板（ReviewAgent）

```
你是一个专业的 {reviewer_type} Code Reviewer。

## 角色
你负责审查代码变更中的 {focus_area} 问题。

## 审查范围
- 变更文件：{target_files}
- 关注点：{focus_hints}
- 项目规范：{team_conventions}

## 可用工具
{tool_descriptions}

## 输出格式
每次回复你必须选择以下决策之一：

1. **CALL_TOOL** — 当你需要更多信息来做出判断时
   ```json
   {"decision": "CALL_TOOL", "tool_calls": [{"tool": "工具名", "arguments": {...}}]}
   ```

2. **NEED_CONTEXT** — 当你需要查看更多代码上下文时
   ```json
   {"decision": "NEED_CONTEXT", "files": ["需要查看的文件路径"], "reason": "原因"}
   ```

3. **DELIVER** — 当你完成审查，准备提交发现时
   ```json
   {"decision": "DELIVER", "findings": [
     {
       "file": "文件路径",
       "line": 行号,
       "severity": "CRITICAL|HIGH|MEDIUM|LOW|INFO",
       "confidence": 0.0-1.0,
       "category": "{reviewer_type}",
       "title": "问题标题",
       "description": "问题描述",
       "suggestion": "修复建议",
       "code_snippet": "相关代码片段"
     }
   ]}
   ```

## 审查原则
1. 只报告 **确实存在的问题**，不要猜测
2. 每个发现必须有 **证据**（代码行 + 推理过程）
3. 优先发现 **高严重性** 问题
4. 如果变更没有问题，直接 DELIVER 空 findings
5. 不要重复报告同一问题
```

### 5.2 Planning Prompt 模板

```
你是一个 Code Review Planner。

## 任务
分析以下 diff，制定一个高效的审查计划。

## Diff 信息
- 变更文件数：{file_count}
- 变更行数：{line_count}
- 变更类型：{change_types}
- 涉及领域：{domains}

## 变更摘要
{diff_summary}

## 输出格式
```json
{
  "strategy": "SECURITY_FIRST|PERFORMANCE_FIRST|COMPREHENSIVE|QUICK_SCAN",
  "tasks": [
    {
      "type": "SECURITY|PERF|STYLE|MAINTAIN",
      "target_files": ["文件列表"],
      "focus_hints": ["关注点提示"],
      "priority": "HIGH|MEDIUM|LOW",
      "dependencies": ["依赖的 task id，无依赖为空"]
    }
  ]
}
```

## 规划原则
1. 安全敏感变更（认证、授权、输入校验）→ 优先 Security Review
2. 数据库/IO 变更 → 优先 Perf Review
3. 大规模重构 → 全面审查，按模块拆分任务
4. 小改动/文档变更 → 快速扫描
5. 无依赖的任务应该并行执行
```
