# CodePilot — AI Code Review Agent

> 一个面向真实研发场景的 AI Code Review Agent，不是框架封装，而是手搓 Agent 工程内核。

---

## 1. 为什么选这个方向

### 1.1 痛点真实

Code Review 是每个研发团队的刚需，但现状堪忧：

- **人工 CR 质量参差不齐**：高级工程师忙，初级工程师看不懂，很多 CR 流于形式
- **速度慢**：一个中等 PR 的 CR 平均等待 4-8 小时，大型 PR 更久
- **重复性问题漏检**：安全漏洞、性能反模式、团队规范违规这类 pattern 化问题，人工容易疲劳漏过
- **知识不沉淀**：每次 CR 的经验随人走，团队没有积累

### 1.2 Agent 工程密度高

Code Review 天然是一个需要"观察-推理-行动"循环的任务，自然覆盖 Agent 核心工程概念：

| Agent 工程概念 | Code Review 中的体现 |
|---|---|
| **ReAct** | Review Agent：观察 diff → 推理潜在问题 → 调用工具验证（查 AST、搜历史）→ 产出 finding |
| **Planning** | Planner 拆分 review 为有序子任务：先看入口变更 → 追踪影响范围 → 专项检查 |
| **Multi-Agent** | Supervisor + Planner + 多个专项 Reviewer（安全/性能/规范/可维护性），可并行 |
| **Tool Use** | AST 解析、Git blame、依赖图查询、历史 review 检索、文件读取 |
| **Context Engineering** | 按 diff 动态编译上下文（只拉相关文件 + 调用链），不塞整个 repo |
| **Memory** | 项目级 review 知识库（历史 pattern、团队规范、常见 bug pattern），跨 PR 学习 |
| **Harness Engineering** | Git Webhook 接入、SSE 流式输出、PR Comment 回写、报告生成 |

### 1.3 可量化、面试好聊

- 准确率 / 误报率 / 召回率：在真实 PR 上跑，对比人工标注
- 效率提升：Agent CR 耗时 vs 人工 CR 平均耗时
- 上下文压缩率：编译后 token 数 vs 全量 token 数
- 专项发现分布：安全漏洞 / 性能问题 / 规范违规各多少

面试时一句话讲清业务价值："AI 帮你做 Code Review，自动发现安全漏洞、性能反模式和规范违规"，面试官秒懂，技术深度可以从 Agent Loop、Context 编译、Memory 系统一路聊下去。

## 2. 与看过项目的对比

### 2.1 诊断

| 看过的项目 | 问题 | CodePilot 怎么解决 |
|---|---|---|
| xfg 系列 | 本质是"组装节点"，框架味重，缺自己的工程思考 | 不用框架高层封装，手搓 Agent Loop 和 Context 编译 |
| ragent / PaiSmart | 纯 RAG，Agent 工程深度不够，简历撑不起亮点 | RAG 只是 Memory 的一部分，重心在 Agent 编排和 Tool Use |
| oncall-agent / ByteCoach | 用框架封装为主，缺可量化数据和差异化 | 可量化评测是核心交付物，不是附加品 |
| ThoughtCoding / AgentX | 方向对（Coding Agent），但 AgentX 太重（294 文件、5 层 DB），ThoughtCoding 又偏简单 | 取中间：比 ThoughtCoding 深（Multi-Agent + Planning + Memory），比 AgentX 轻（不做代码生成，只做审查） |
| nanobot / gemini-cli | 好参考但不是 Java，直接改容易"四不像" | 借鉴架构模式（AgentLoop/Runner 分离、Context 治理、Tool 并发），用 Java 重新实现 |

### 2.2 从开源项目借鉴什么

| 开源项目 | 借鉴点 | 在 CodePilot 中的应用 |
|---|---|---|
| **nanobot** | AgentLoop / AgentRunner 分离；MessageBus 解耦；Context 治理（Microcompact、History Snipping）；Tool 并发分区（只读并行、互斥串行） | Orchestrator 对应 AgentLoop，ReviewEngine 对应 AgentRunner；事件驱动解耦前端和 Agent 内核；上下文治理策略直接用 |
| **gemini-cli** | Policy Engine（工具审批策略）；Hook 生命周期；Loop Detection（检测 Agent 死循环）；Chat Compression | Tool 执行策略（只读工具自动执行、写操作需确认）；Review 流程 Hook（before/after review）；死循环检测防止 Agent 无限分析 |
| **AgentX** | 固定 Workflow 思想；Context Compilation Center；Stack Profile 扩展机制；Eval Center | 固定 Review Pipeline（diff→plan→review→merge→report）；上下文编译是核心模块；语言/框架 Profile 支持扩展；评测中心产出可量化数据 |
| **ThoughtCoding** | MCP 工具扩展模式；非 Function Calling 模型的 Prompt 驱动工具调用 | 工具以 MCP 方式扩展；支持不原生支持 Function Calling 的模型 |

## 3. 核心设计理念

### 3.1 只读审查，不做代码生成

Code Review Agent 只"读"和"分析"，不"写"代码。这带来三个好处：

1. **安全边界清晰**：Agent 不会修改你的代码，只有建议权
2. **难度可控**：代码生成是 Coding Agent 最难的部分（AgentX 用了 Docker + Git Worktree 隔离），审查只需要理解代码
3. **验证更容易**：审查结果的对错可以人工快速判断，代码生成的好坏需要跑测试

### 3.2 手搓内核，不用框架封装

核心 Agent Loop、Context 编译、Tool 调度、Memory 管理**全部自己实现**，这是简历上真正能聊出深度的东西。框架（Spring AI / LangChain4j）只在模型调用层做薄封装。

### 3.3 上下文编译是第一公民

借鉴 AgentX 的 Context Compilation Center 思想：上下文不是"把文件塞进 prompt"，而是一个专门的编译过程，按 diff 动态决定拉什么、切什么、排什么。

### 3.4 可评测、可量化

评测不是事后补充，是架构的一等公民。每个模块都有可观测的输入输出，Eval Center 可以端到端跑评测产出 scorecard。
