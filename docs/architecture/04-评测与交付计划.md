# CodePilot — 评测体系与交付计划

---

## 1. 评测体系设计

评测不是事后补充，是架构的一等公民。借鉴 AgentX 的 Eval Center 思想。

### 1.1 评测指标

| 指标 | 定义 | 数据来源 | 目标 |
|------|------|---------|------|
| **Precision** | 报告的问题中真正有问题的比例 | 人工标注 | ≥ 0.7 |
| **Recall** | 真实问题中被 Agent 发现的比例 | 人工标注 | ≥ 0.6 |
| **F1 Score** | Precision 和 Recall 的调和平均 | 计算 | ≥ 0.65 |
| **False Positive Rate** | 误报比例 | 人工标注 | ≤ 0.3 |
| **Review 延迟** | 从 PR 创建到 review 完成的时间 | 系统日志 | ≤ 5min (中小 PR) |
| **Token 效率** | ContextPack tokens / 全量 tokens | ContextCompiler | ≤ 0.3 |
| **Loop Detection 准确率** | 正确检测死循环 / 总检测次数 | LoopDetector 日志 | ≥ 0.8 |
| **Memory 命中率** | Memory 检索结果被 Agent 引用的比例 | ReviewSession 日志 | ≥ 0.5 |
| **端到端成功率** | 无错误完成 review 的比例 | Session 日志 | ≥ 0.9 |

### 1.5 Baseline 对比

与以下 baseline 对比，展示 CodePilot 的差异化：

| Baseline | 描述 |
|----------|------|
| **直接 LLM Review** | 把 diff 直接喂给 LLM，不加工具、不加上下文编译、不加记忆 |
| **Lint Only** | 只用静态分析工具（SpotBugs、Checkstyle），不加 LLM |
| **LLM + Full Context** | 塞整个 repo 上下文给 LLM，不做编译优化 |
| **CodePilot** | 完整系统：Planning + Context 编译 + Multi-Agent + Memory + Tools |

预期结论：CodePilot 在 recall 上远超 Lint Only，在 precision 上远超直接 LLM Review，在 token 效率上远超 Full Context。

## 2. 分期交付计划

### Phase 1: MVP — 单 Agent 跑通（第 1-3 周）

**目标**：最短路径让一个 PR 被 review 并回写评论。

```
Week 1: 基础设施
  - 项目骨架搭建（Maven 多模块）
  - LLM 调用层实现（OpenAI 兼容协议 + 流式）
  - GitHub Webhook 接收 + PR Comment 回写
  - CLI 模式：本地 review 一个 diff 文件

Week 2: Agent 内核
  - ReviewEngine (Agent Loop) 实现
  - Tool 抽象 + ToolRegistry + ToolExecutor
  - 基础工具：read_file, ast_parse, search_pattern
  - Prompt 模板 + 结构化输出解析
  - 单 ReviewAgent 跑通：输入 diff → 输出 findings

Week 3: 上下文 + 集成
  - ContextCompiler 基础版（diff 分析 + 文件拉取）
  - 整合：Webhook → 单 Agent Review → PR Comment
  - 端到端 demo：提交一个有 SQL 注入的 PR → Agent 自动 review
```

**交付物**：
- 能跑的 demo（CLI + Webhook 两种模式）
- 至少能发现 SQL 注入、空指针等基础问题
- 架构博客 / README（记录设计思考）

### Phase 2: Multi-Agent + Planning（第 4-6 周）

**目标**：体现 Agent 编排能力，体现工程思考。

```
Week 4: Planning
  - PlanningAgent 实现
  - ReviewPlan + TaskGraph 建模
  - 按 diff 特征选择 ReviewStrategy
  - 任务依赖排序 + 并行调度

Week 5: Multi-Agent
  - ReviewAgent Pool：Security / Perf / Style / Maintainability
  - Orchestrator 状态机 + 任务调度
  - 并行执行无依赖的 Reviewer
  - MergeAgent：合并 findings、去重排序
  - SSE 流式推送 review 进度

Week 6: 增强
  - Loop Detection（双模式）
  - Context Governance（Microcompact + History Snipping）
  - Token 预算管理
  - 工具并发分区（只读并行）
  - 更丰富的工具：git_blame, ast_find_references, ast_get_call_chain
```

**交付物**：
- Multi-Agent 协同 review demo
- SSE 实时进度推送
- 与单 Agent 的 ablation 对比数据

### Phase 3: Memory + 评测（第 7-9 周）

**目标**：体现 Memory 系统和可量化评测，打造简历亮点。

```
Week 7: Memory
  - Project Memory 存储（MySQL + PgVector）
  - Team Convention 管理
  - Memory 检索融入 Context 编译
  - Dream 两阶段沉淀（review 后自动学习）

Week 8: 评测
  - Eval Scenario 编写（47 个场景）
  - EvalRunner 实现
  - Scorecard 生成
  - Baseline 对比实验
  - 评测数据可视化

Week 9: 打磨
  - 错误处理 + 优雅降级
  - Session 事件溯源 + 中断恢复
  - 文档完善
  - 简历亮点提炼
  - 技术博客撰写
```

**交付物**：
- 完整评测报告（Scorecard + Baseline 对比）
- 项目级 Memory 效果展示
- 简历项目描述 + 面试话术
- GitHub README + 架构图

## 3. 简历呈现建议

### 3.1 项目描述（简历上）

> **CodePilot — AI Code Review Agent**
>
> 面向真实研发场景的 AI Code Review 系统，通过 Multi-Agent 协同 + Context 编译 + 项目记忆，自动审查 PR 中的安全漏洞、性能反模式和规范违规。
>
> - 设计并实现 ReAct Agent Loop，支持 Tool 并发分区（只读并行/互斥串行）、Loop Detection（pattern + LLM 双模式）、Context Governance（Microcompact + History Snipping）
> - 实现 Context Compilation Center：基于 AST 依赖图动态编译 review 上下文，token 效率 0.24（vs 全量 1.0），确保关键上下文不被裁剪
> - 实现 Multi-Agent 编排：Planner 分析 diff 生成 ReviewPlan（任务 DAG），4 个专项 Reviewer 并行审查，Merger 合并去重
> - 实现三层 Memory 系统 + Dream 两阶段沉淀，项目级 pattern 跨 PR 学习，memory 命中率 51%
> - 在 47 个评测场景上端到端评测：F1=0.67，review 延迟 182s（vs 人工 CR 平均 4h+），误报率 27%

### 3.2 面试聊点

| 聊点 | 深度问题 |
|------|---------|
| **Agent Loop** | "你的 Agent Loop 和 LangChain4j 的 AiServices 有什么区别？" → 我自己实现决策解析、Loop Detection、Context Governance，AiServices 是黑盒 |
| **Context 编译** | "为什么不直接把整个 repo 塞给 LLM？" → token 有限 + 无关信息干扰推理；展示 AST 感知的选择策略和 token 预算分配 |
| **Multi-Agent** | "为什么不用一个 Agent 全干？" → 单 Agent context 太长容易遗漏；专项 Reviewer 聚焦 + 并行更快；ablation 数据证明 |
| **Memory** | "项目记忆和 RAG 有什么区别？" → RAG 是被动检索，Memory 是主动学习 + 沉淀；Dream 机制让系统越用越准 |
| **Loop Detection** | "Agent 死循环怎么处理？" → 双模式：pattern 检测快速捕获常见循环 + LLM 判断处理复杂场景；降级策略 |
| **可量化** | "你的系统到底好不好用？" → 47 场景评测 + baseline 对比；不是"感觉好用"而是有数据 |

| 烂大街项目 | CodePilot |
|-----------|-----------|
| "用 LangChain4j 做了一个 RAG 问答" | "手搓 Agent Loop + Context 编译 + Multi-Agent 编排" |

## 4. 风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| **LLM 输出不稳定** | 同一 diff 多次 review 结果不同 | 评测取 3 次平均值；prompt 中加"只报告确定性发现"约束 |
| **AST 解析失败** | 非标准 Java 代码、Lombok 等 | 降级为正则匹配 + 文本分析；缓存解析结果 |
| **Token 超限** | 大 PR 导致编译后上下文仍超限 | Context Governance 兜底；大 PR 按模块拆分 |
| **GitHub API 限流** | PR Comment 回写被限流 | 指数退避重试；批量 comment 合并为 review |
| **评测场景偏差** | 手工构造的场景不代表真实情况 | 补充 3-5 个开源项目的真实 PR 作为评测场景 |
| **做不完** | 3 期 9 周可能超期 | Phase 1 即可 demo；Phase 2 是核心差异化；Phase 3 是锦上添花。每期独立可用 |
