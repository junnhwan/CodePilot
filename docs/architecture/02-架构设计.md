## 3. Agent 设计

### 3.1 Agent 角色定义

```
┌─────────────────────────────────────────────────┐
│              Orchestrator (Supervisor)           │
│  职责：会话管理、任务调度、流控、Hook 执行       │
│  决策：DISPATCH / WAIT / ESCALATE / COMPLETE     │
└────────────┬──────────────────┬─────────────────┘
             │                  │
    ┌────────▼────────┐  ┌──────▼──────────┐
    │ PlanningAgent   │  │  MergeAgent      │
    │ 输入：diff      │  │  输入：所有      │
    │ 输出：ReviewPlan│  │  findings        │
    │                 │  │  输出：ReviewResult│
    └────────┬────────┘  └──────▲──────────┘
             │                  │
    ┌────────▼──────────────────┴──────────┐
    │         ReviewAgent (专项 Reviewer)    │
    │                                       │
    │  SecurityReviewer  │  注入、越权、泄露 │
    │  PerfReviewer      │  N+1、内存泄漏、O │
    │  StyleReviewer     │  规范、命名、格式 │
    │  MaintainReviewer  │  复杂度、耦合、重 │
    │                                       │
    │  共同接口：                           │
    │  输入：ReviewTask + ContextPack       │
    │  输出：List<Finding>                  │
    │  决策：CALL_TOOL / ASK_CONTEXT / DONE │
    └───────────────────────────────────────┘
```

### 3.2 Agent 状态机

```
         Webhook 触发
              │
              ▼
         ┌────────┐
         │  IDLE  │
         └───┬────┘
             │ diff 到达
             ▼
      ┌─────────────┐
      │  PLANNING   │  PlanningAgent 分析 diff，产出 ReviewPlan
      └──────┬──────┘
             │ plan 就绪
             ▼
      ┌─────────────┐
      │  REVIEWING  │  ReviewAgent Pool 按计划执行 review
      └──────┬──────┘
             │ 所有 task 完成
             ▼
      ┌─────────────┐
      │  MERGING    │  MergeAgent 合并 findings，去重排序
      └──────┬──────┘
             │ report 生成
             ▼
      ┌─────────────┐
      │  REPORTING  │  回写 PR Comment，更新 Memory
      └──────┬──────┘
             │
             ▼
      ┌─────────────┐
      │   DONE      │
      └─────────────┘
```

异常路径：
- **TIMEOUT**：单个 ReviewTask 超时 → Orchestrator 标记为 SKIPPED，继续其余任务
- **STUCK**：Agent 死循环检测（借鉴 gemini-cli 的 Loop Detection）→ 强制终止当前 turn，降级为简单 review
- **LLM_FAILURE**：模型调用失败 → 重试（带 429 分类，借鉴 nanobot），超过 3 次跳过该 task
- **INTERRUPTED**：服务崩溃 → Session 事件溯源，恢复时从最后 checkpoint 继续

### 3.3 Agent Loop 核心流程

每个 ReviewAgent 的内部执行遵循统一的 ReAct Loop：

```java
// 伪代码，展示核心逻辑
class ReviewEngine {
    ReviewTaskResult execute(ReviewTask task, ContextPack context) {
        var messages = buildInitialPrompt(task, context);
        var iterations = 0;

        while (iterations < MAX_ITERATIONS) {
            // 1. 调用 LLM
            var response = llm.chat(messages);

            // 2. 解析决策
            var decision = parseDecision(response);

            switch (decision.type()) {
                case CALL_TOOL -> {
                    // 3. 执行工具（只读并行，互斥串行）
                    var results = toolExecutor.execute(decision.toolCalls());
                    // 4. 工具结果注入上下文
                    messages.addAll(results.toMessages());
                }
                case ASK_CONTEXT -> {
                    // 5. 请求更多上下文（Context Compiler 按需拉取）
                    var extra = contextCompiler.compile(decision.contextRequest());
                    messages.add(extra.toMessage());
                }
                case DONE -> {
                    // 6. 提取 findings
                    return extractFindings(response);
                }
            }

            iterations++;

            // 7. Loop Detection
            if (loopDetector.isStuck(messages)) {
                return degradeToFallback(task, context);
            }

            // 8. Context Governance：token 预算检查
            if (tokenCounter.count(messages) > TOKEN_BUDGET) {
                messages = contextGovernor.compact(messages);
            }
        }

        return ReviewTaskResult.timeout(task);
    }
}
```

关键设计点：
- **不依赖框架的 Function Calling**：通过 Prompt + 结构化输出解析实现工具调用，兼容任何模型（借鉴 ThoughtCoding）
- **Tool 并发分区**：只读工具（AST 查询、文件读取、历史搜索）并行执行，互斥工具串行（借鉴 nanobot）
- **Context Governance**：Microcompact（旧工具结果摘要化）+ History Snipping（按 token 预算裁剪历史）
- **Loop Detection**：简单模式（检测连续 3 次相同 tool call）+ LLM 模式（用小模型判断是否陷入循环，借鉴 gemini-cli）

## 4. Context Compilation 设计

这是 CodePilot 最核心的工程差异化——上下文不是"塞文件"，而是"编译"。

### 4.1 编译流程

```
  Diff (changed files + hunks)
              │
              ▼
  ┌───────────────────────┐
  │ Step 1: DiffAnalysis  │  解析变更：哪些文件、哪些方法、变更类型
  │ 输出：DiffSummary      │  (add/modify/delete, 方法级粒度)
  └───────────┬───────────┘
              │
              ▼
  ┌───────────────────────┐
  │ Step 2: ImpactScope   │  基于 AST 依赖图，计算影响范围
  │ 输出：ImpactSet        │  (受影响的类/方法/调用链)
  └───────────┬───────────┘
              │
              ▼
  ┌───────────────────────┐
  │ Step 3: FileSelection │  按 ImpactSet + Token 预算，选择要拉取的文件
  │ 输出：FileManifest     │  (优先级排序的文件列表)
  └───────────┬───────────┘
              │
              ▼
  ┌───────────────────────┐
  │ Step 4: ContentFetch  │  读取文件内容，AST 解析，提取符号表
  │ 输出：RawContext       │  (文件内容 + 符号信息)
  └───────────┬───────────┘
              │
              ▼
  ┌───────────────────────┐
  │ Step 5: ContextPack   │  组装最终上下文：结构化事实 + 代码片段 + 记忆
  │ 输出：ContextPack      │  (token 预算内的精编译码上下文)
  └───────────────────────┘
```

### 4.2 ContextPack 结构

```java
record ContextPack(
    // 结构化事实：运行环境、语言、框架
    StructuredFacts facts,
    // 变更摘要：diff 的结构化表示
    DiffSummary diff,
    // 影响范围：受影响的类/方法/调用链
    ImpactSet impact,
    // 代码片段：按优先级选取的文件内容
    List<CodeSnippet> snippets,
    // 项目记忆：相关历史 pattern、规范
    List<MemoryEntry> memories,
    // Token 统计
    TokenBudget budget
) {}
```

### 4.3 关键策略

| 策略 | 说明 | 量化价值 |
|------|------|---------|
| **按需拉取** | 只拉 diff 涉及的文件 + 直接依赖，不拉整个 repo | token 节省 60-80% |
| **AST 感知** | 解析到方法级粒度，只塞变更方法的签名 + 上下文，而非整个文件 | token 节省 40-60% |
| **优先级排序** | 变更文件 > 直接调用方 > 间接依赖 > 配置文件 > 测试文件 | 保证关键上下文不被裁剪 |
| **动态预算** | 根据模型 token 上限和 diff 大小动态分配各部分预算 | 不同模型自适应 |
| **增量编译** | 同一 PR 的多次 review（push 新 commit）复用上次编译结果 | 重复 review 速度提升 3-5x |

## 5. Memory 系统设计

### 5.1 记忆分层

```
┌────────────────────────────────────┐
│  Layer 1: Working Memory           │  当前 review 会话的上下文
│  存储：内存 (Session-scoped)        │  生命周期：单次 review
│  内容：当前 diff、编译的上下文、    │
│        已产出的 findings            │
└────────────────────────────────────┘
         │ review 完成后沉淀
         ▼
┌────────────────────────────────────┐
│  Layer 2: Project Memory           │  项目级知识
│  存储：PgVector + MySQL            │  生命周期：项目级持久
│  内容：                             │
│   - ReviewPattern: 历史发现的问题   │
│     pattern (如"这个项目常犯N+1")  │
│   - TeamConvention: 团队编码规范    │
│     (如"这个团队用 Slf4j 不用 Log4j")│
│   - CodeStyle: 项目的代码风格偏好   │
└────────────────────────────────────┘
         │ 跨项目泛化
         ▼
┌────────────────────────────────────┐
│  Layer 3: Global Knowledge         │  通用安全/性能知识库
│  存储：内置规则 + 向量检索           │  生命周期：永久
│  内容：                             │
│   - SecurityPatterns: SQL注入、XSS │
│     等通用安全 pattern              │
│   - PerfAntiPatterns: O(n²)循环、  │
│     内存泄漏等通用性能反模式        │
└────────────────────────────────────┘
```

### 5.2 Memory 检索流程

```
ReviewTask 到达
       │
       ▼
```

## 6. Planning 设计

### 6.1 ReviewPlan 结构

```java
record ReviewPlan(
    String planId,
    String sessionId,
    DiffSummary diff,
    // 任务 DAG：有依赖关系的任务图
    TaskGraph taskGraph,
    // 全局策略：优先关注什么
    ReviewStrategy strategy
) {}

record ReviewTask(
    String taskId,
    TaskType type,          // SECURITY / PERF / STYLE / MAINTAIN
    List<String> targetFiles,
    List<String> focusHints, // Planner 给的提示
    List<String> dependencies, // 依赖的 task id
    Priority priority
) {}

enum ReviewStrategy {
    SECURITY_FIRST,    // 安全优先：有安全敏感变更时
    PERFORMANCE_FIRST, // 性能优先：有数据库/IO变更时
    COMPREHENSIVE,     // 全面审查：大型重构
    QUICK_SCAN         // 快速扫描：小改动/文档变更
}
```

### 6.2 Planner 决策逻辑

```
Diff 到达
    │
    ▼
┌──────────────┐
│ Diff 分类     │  变更类型：新增/修改/删除
│              │  变更规模：行数、文件数
│              │  变更领域：安全敏感？IO密集？纯样式？
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 策略选择      │  根据分类结果选择 ReviewStrategy
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 任务生成      │  按策略生成 ReviewTask 列表
│              │  - 大 diff → 按文件/模块拆分
│              │  - 安全敏感文件 → SecurityReviewer
│              │  - 数据库相关 → PerfReviewer
│              │  - 无明显侧重 → 全部 Reviewer
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 依赖建模      │  建立任务间依赖
│              │  - 公共上下文任务先执行
│              │  - 无依赖的 Reviewer 并行
│              │  - 有依赖的串行
└──────┬───────┘
       │
       ▼
  ReviewPlan → Orchestrator 调度
```

## 7. Tool 系统设计

### 7.1 Tool 抽象

```java
interface Tool {
    String name();
    String description();
    JsonSchema parameters();      // JSON Schema 描述参数
    boolean readOnly();           // 只读工具可并行
    boolean exclusive();          // 互斥工具必须串行

    ToolResult execute(ToolCall call, ToolContext ctx);
}

record ToolCall(
    String callId,
    String toolName,
    Map<String, Object> arguments
) {}

record ToolResult(
    String callId,
    boolean success,
    String output,               // 文本输出
    Map<String, Object> metadata // 结构化元数据
) {}
```

### 7.2 内置工具

| 工具名 | 类型 | 用途 | 只读 |
|--------|------|------|------|
| `ast_parse` | AST | 解析 Java 文件，返回符号表、方法签名、类型信息 | 是 |
| `ast_find_references` | AST | 查找某个类/方法的所有引用 | 是 |
| `ast_get_call_chain` | AST | 获取某个方法的调用链（向上/向下） | 是 |
| `git_blame` | Git | 查看某行代码的最后修改者和时间 | 是 |
| `git_log` | Git | 查看某个文件的提交历史 | 是 |
| `git_diff_context` | Git | 获取变更的完整上下文（前后 N 行） | 是 |
| `read_file` | IO | 读取指定文件内容 | 是 |
| `search_pattern` | IO | 在项目中搜索代码 pattern（正则） | 是 |
| `memory_search` | Memory | 检索项目记忆（历史 review pattern） | 是 |
| `memory_lookup_rule` | Memory | 查找团队规范 | 是 |
| `web_search` | Web | 搜索外部知识（如 CVE 库、API 文档） | 是 |

### 7.3 Tool 执行策略

```
Agent 产出多个 ToolCall
         │
         ▼
  ┌───────────────┐
  │ 分区           │  按 readOnly + exclusive 属性分区
  │  只读区：并行   │  ast_parse, read_file, memory_search 等
  │  互斥区：串行   │  （当前场景下全部只读，无互斥工具）
  └───────┬───────┘
          │
          ▼
  ┌───────────────┐
  │ 并行执行       │  CompletableFuture.allOf() 执行只读区
  │  超时控制       │  每个工具调用 30s 超时
  │  结果缓存       │  相同参数的工具调用缓存结果
  └───────┬───────┘
          │
          ▼
  ToolResult[] → 注入 Agent 上下文
```

## 8. 数据流：端到端一次 Review

```
1. 开发者提交 PR
2. GitHub Webhook → Gateway 接收
3. Gateway 鉴权 + 去重 → 创建 ReviewSession
4. SSE 推送：session_created
5. Orchestrator 启动 Planning Agent
   5.1 Planning Agent 分析 diff
   5.2 Planning Agent 产出 ReviewPlan (task DAG)
6. SSE 推送：plan_ready
7. Context Compiler 编译上下文
   7.1 AST 解析变更文件
   7.2 计算影响范围
   7.3 拉取相关文件
   7.4 检索项目记忆
   7.5 组装 ContextPack
8. Orchestrator 按计划调度 Review Agents
   8.1 无依赖的 Reviewer 并行执行
   8.2 每个 Reviewer 执行 ReAct Loop
   8.3 SSE 推送：task_started / finding_found / task_completed
9. Merge Agent 合并所有 findings
   9.1 去重（相同位置不同 Reviewer 的发现合并）
   9.2 排序（按严重等级 + 置信度）
   9.3 过滤（低置信度 findings 降级或移除）
10. Report Engine 生成报告
    10.1 Markdown 格式的 Review Summary
    10.2 PR Inline Comments (逐文件逐行)
11. Gateway 回写 GitHub PR
    11.1 POST /repos/{owner}/{repo}/pulls/{number}/reviews
    11.2 POST /repos/{owner}/{repo}/pulls/{number}/comments
12. Memory 沉淀（异步）
    12.1 Dream Phase 1：分析 review 结果
    12.2 Dream Phase 2：增量更新项目记忆
13. SSE 推送：review_completed
14. Session 归档
```

## 9. 与 AgentX 的架构对比

| 维度 | AgentX | CodePilot | 设计取舍 |
|------|--------|-----------|---------|
| Workflow | 固定 8 步管线 | 固定 6 步管线 | 更短，因为只做审查不做代码生成 |
| Agent 数量 | 4 个（需求/架构/编码/验证） | 3 类（Planner/Reviewer/Merger） | Reviewer 可扩展，按专项拆分 |
| 状态持久化 | MySQL 5 层 schema | MySQL + 事件溯源 | 简化：不需要 L1-L5 全套，但保留事件溯源能力 |
| 执行隔离 | Docker + Git Worktree | 无需隔离（只读审查） | 大幅降低基础设施复杂度 |
| 上下文编译 | Context Compilation Center | 同理但更轻量 | 不需要 Stack Profile（先只做 Java），但保留扩展点 |
| 评测 | Eval Center + Scenario Pack | 同理 | 核心评测能力保留，场景可简化 |
| Token 预算管理 | 无显式管理 | 显式 TokenBudget + Context Governance | 这是我们的差异化 |
| Loop Detection | 无 | 双模式（pattern + LLM） | 借鉴 gemini-cli |
| Memory | 无项目级记忆 | 三层 Memory + Dream 沉淀 | 这是我们的差异化 |

**核心差异化**：Context Engineering（上下文编译 + token 预算治理）和 Memory（项目级记忆沉淀）是 CodePilot 相比 AgentX 的两大工程亮点，也是面试中最能聊出深度的部分。
